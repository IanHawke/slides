METHOD

Alex Wright's testbed code.

  • FDTD, 2d, serial or GPU, OMP (ish);
  • Google test framework;
  • not MPI capable;
  • plain text output (and no checkpoint/restart);

Want

  1. MPI+GPU;
  2. HDF5 output with checkpoint restart.

Testbed has been very useful, especially for checking GPU performance of novel algorithms. Memory limited without MPI. Queue time limited without checkpoint/restart. Diskspace limited without HDF5 output.

Einstein Toolkit

Production code, used by all.

  • FDTD, 3d, OMP+MPI (not GPU), AMR;
  • big, long simulations;
  • many compilation/library combinations;
  • HDF5 output in odd format.

Want

  1. How can the code be more efficient on Iridis? Compile flags, memory layout, library choices, etc.
  2. Better viz pipeline, through eg yt. Current scripts simple, slow as recombining files. Community scripts not easy to use.

It will be difficult to optimize the code directly; lots of interconnected algorithms, probably limited by the AMR communication model. Probably get the main gain by asking how we can get the most out of Iridis. The viz pipeline is hopefully pure Python work, looking to speed up exploratory viz (although we should discuss Visit). The problems with the community scripts are largely around dependencies and many are based on Python 2.
